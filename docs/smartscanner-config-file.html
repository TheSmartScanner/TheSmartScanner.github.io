SmartScanner Configuration File Guide

SmartScanner supports loading a configuration file through the CLI using the -c or --config flag. This file allows you to fully customize scan behavior, HTTP settings, crawling rules, excluded file types, form input rules, vector configuration, and more.

This page explains the full structure of the configuration file using the default configuration (default-scan-config.json) as a reference.

1. Version

"version": "1.0"

Specifies the configuration format version.

2. Tests

The tests array lists vulnerability tests or test groups to execute.

"tests": ["SQL Injection/?error&blind&time", "OS Command Execution", ...]

Tests may be:

  • Individual tests (e.g., OS Command Execution)
  • Grouped tests using syntax like: TestName/?option1&option2

3. HTTP Settings

HTTP settings control how SmartScanner makes requests during a scan. These options directly map to the UI settings found under Scan Config → Advanced Settings → HTTP Tab.

"http": {
  "timeout": 60000,
  "maxParallelRequests": 4,
  "maxRequestsPerSecond": false,
  "headers": [...],
  "cookies": [],
  "userAgent": "Mozilla/5.0 (Windows NT 10.0; Win64; x64)..."
}

Timeout (seconds) → timeout

How long SmartScanner waits before giving up on a request.

  • UI value is in seconds.
  • Config file uses milliseconds.
  • Example: UI input 30 → config value 30000.

Parallel requests → maxParallelRequests

How many simultaneous HTTP requests SmartScanner will send.

  • Higher values speed up scans.
  • May increase server load.

Max requests/second → maxRequestsPerSecond

Rate-limits outgoing requests.

  • false = disabled.
  • A number = requests per second.
  • UI uses checkbox + number; config uses a single field.

User agent → userAgent

The User-Agent string sent with all HTTP requests.

  • Can be a browser signature or a custom string.
  • UI provides presets; config stores the full string.

Additional Cookies → cookies

Cookies added to every request. Each cookie is stored as:

{"name": "CookieName", "value": "CookieValue"}

Additional HTTP Headers → headers

Custom headers sent with every request. Each header entry is:

{"name": "Header-Name", "value": "header value"}

4. Crawler Settings

Crawler settings control how SmartScanner explores (crawls) the target website. These map to the UI options under Scan Config → Advanced Settings → Crawler Tab.

"crawler": {
  "scope": {
    "crawlStartDepth": 0,
    "type": "auto",
    "scanSubDomains": false,
    "scanAbovePath": false
  },
  "evaluteJsWithChromium": true,
  "depth": false,
  "count": 70000,
  "fileExclusion": "*.7z,*.zip,...",
  "urlExclusion": ["*sign?out*", "*.svn/+", ...]
}

Maximum Depth → depth

Controls how many link-levels the crawler follows.

  • UI uses checkbox + number.

  • Config uses:

    • false = unlimited depth
    • number = max depth
  • 0 means scan only the user-provided URLs.

Maximum Count → count

Limits the total number of URLs the crawler can visit.

  • UI uses checkbox + number.
  • Config stores the number directly.
  • Prevents extremely large scans (default: 70,000).

File Exclusion → fileExclusion

Skips URLs whose file name matches patterns.

  • UI: multiline pattern text (e.g., *.css, debug*).
  • Config: comma-separated list string.
  • Applied to filename only, not full path.

Evaluate JavaScripts Using Chromium → evaluteJsWithChromium

Enables headless Chromium rendering.

  • Required for SPAs or dynamically generated links.

Scope (Smart / Manual) → scope

Defines crawl boundaries.

Smart Mode (type: "auto")

  • scanSubDomains: include subdomains.
  • scanAbovePath: allow crawling to paths above the target’s directory.

Manual Mode (type: "manual")

  • You must supply a regex for scope restriction.
  • Only URLs matching the regex will be crawled.

URL Exclusion → urlExclusion

Patterns that prevent visiting or testing specific URLs.

  • UI uses one pattern per line.
  • Config uses an array of strings.
  • Empty lines are ignored.

Pattern Language (matching UI rules)

Supported wildcard tokens:

  • * → 0 or more characters
  • + → 1 or more characters
  • ? → any single character
  • % → 1+ chars except / and ?
  • # → digit
  • Special tokens: [md5], [guid], [year], [base64], [seo], [non-english]

These are valid in both URL Exclusion and test rules.

Practical Examples

  • Shallow scan: set depth to 2.
  • Cap large site: set count to 1000.
  • Skip static assets: add /assets/*, /images/*, *.css to urlExclusion.

5. Proxy

SmartScanner supports scanning websites through different types of proxies. These settings map directly to the UI options under Scan Config → Proxy Tab.

"proxy": {
  "type": "system",
  "host": "",
  "port": 0,
  "user": "",
  "pass": ""
}

System Proxy → type: "system"

Uses the operating system’s proxy configuration.

  • No additional fields required.

HTTP Proxy → type: "http"

Used for traditional HTTP/HTTPS proxy servers.

  • Requires:

    • host: hostname or IP
    • port: port number

  • Optional:

    • user
    • pass

Minimal config example:

"proxy": {
  "type": "http",
  "host": "192.168.0.10",
  "port": 8080,
  "user": "",
  "pass": ""
}

SOCKS Proxy (Recommended) → type: "socks5"

Supports SOCKS5 (IPv4/IPv6).

  • Requires:

    • host
    • port

  • Optional:

    • user
    • pass

Example:

"proxy": {
  "type": "socks5",
  "host": "10.0.0.5",
  "port": 1080,
  "user": "proxyuser",
  "pass": "proxypass"
}

Notes

  • SmartScanner does not use any proxy unless explicitly configured.
  • Leave username and password empty if your proxy doesn’t require authentication.
  • SOCKS5 is recommended for better compatibility and performance.

6. Authentication

SmartScanner supports two main authentication methods: HTTP Basic Authentication and Form-Based Authentication. These allow scanning of protected areas after logging in. The CLI configuration maps directly to UI options found under Scan Config → Authentication Tab.

"authentication": {
  "manualLogin": false,
  "http": []
}

HTTP Basic Authentication → authentication.http

Used when the target website uses built-in HTTP Basic Auth.

  • UI: select HTTP, enter username & password.
  • Config file: stored as authentication entries inside the http array.
  • CLI alternative: --auth-basic user:pass.

Minimal Config File Requirements

  • To enable HTTP Basic Auth via config:
"authentication": {
  "manualLogin": false,
  "http": [{ "user": "admin", "pass": "secret" }]
}

7. Form Input Rules

Form input rules let SmartScanner automatically fill web forms during scanning. These values simulate realistic user inputs and help SmartScanner test form-based vulnerabilities. They map to the UI settings under Scan Config → Advanced Settings → Form Inputs Tab.

"form": {
  "inputs": [
    "*;;*;;email;;*;;test@none.nowhere.com",
    "*;;*;;password;;*;;DJrLcmno321@!",
    "*;;*;;first*name*;;*;;James",
    ...
  ]
}

Purpose

SmartScanner uses these values to:

  • Populate login fields
  • Fill registration or profile fields
  • Submit search forms
  • Trigger validation logic
  • Improve scanning of pages that require user input

The config file stores these as structured rules using SmartScanner’s form rule syntax:

<urlPattern>;;<formPattern>;;<fieldPattern>;;<fieldType>;;<value>

Minimal example:

"form": {
  "inputs": [
    "*;;*;;email;;*;;user@example.com",
    "*;;*;;username;;*;;testuser",
    "*;;*;;password;;*;;Password123"
  ]
}

To simulate a registered user:

"form": {
  "inputs": [
    "*;;*;;username;;*;;testuser",
    "*;;*;;password;;*;;Password123",
    "*;;*;;email;;*;;testuser@example.com"
  ]
}

SmartScanner will use these values in matching form fields across the scanned site.

8. Vector Configuration

Defines how SmartScanner injects payloads.

"vector": {
  "vectors": ["GET", "POST", "Cookie", "Header", "Path"],
  "parameterExclusion": ["*;;PHPSESSID;;Any;;*"]
}

Keys:

  • vectors: Attack surfaces to use.
  • parameterExclusion: Avoids injecting into specific parameters.

Format:

<urlPattern>;;<paramName>;;<method>;;<payloadRule>

9. Technologies

SmartScanner can optimize scanning by enabling technology-specific tests based on the target environment. These settings map to the UI options under Scan Config → Technologies Tab.

"technologies": []

Selecting the technologies used by your target application:

  • Improves accuracy by running only relevant tests
  • Reduces scanning time
  • Avoids unnecessary requests
  • Enables deeper checks for specific platforms (WordPress, Apache, PHP, etc.)

In the configuration file, each technology is represented as an object:

"technologies": [
  { "name": "WordPress", "path": "/blog/" },
  { "name": "Apache", "path": "" },
  { "name": "MySQL", "path": "" }
]
  • name: The technology name (e.g., “Apache”, “WordPress”, “Laravel”, “MySQL”).

  • path: URL path where the technology is used.

    • Empty string = applies globally to the entire target
    • Example: For WordPress installed at https://example.com/blog/, use "/blog/"

If your site uses WordPress at /blog/:

"technologies": [
  { "name": "WordPress", "path": "/blog/" }
]

10. Target

Reserved for future use.

"target": {}

Example Usage with CLI

Load this configuration file:

sms.exe -u https://example.com -c default-scan-config.json
